1กลยุทธ์การขึ้นระบบจริง (Go-Live Strategy)

1.1 รูปแบบที่เลือก — ระบุรูปแบบการขึ้นระบบและเหตุผล (งวดที่ 2 = รพ.หลัก; สาขาทยอยในงวดที่ 3)

รูปแบบเลือกคำอธิบาย / ความเหมาะสมกับ HIS เมืองสมุทร
Big Bang (สับเปลี่ยนทั้ง รพ.หลัก ในคราวเดียว)☐ เลือกตัด HosXP แล้วเปิด HA.OS ที่ รพ.หลัก ในหน้าต่างเวลาเดียว ลดภาระ dual-run

Parallel Run (เดินคู่ขนานชั่วคราว)☐ เลือกบันทึกทั้งสองระบบช่วงสั้น ๆ เพื่อเทียบผล — ใช้ทรัพยากรสูง

  • ช่วงปิดบริการ/ลดบริการ (Downtime): __ ชั่วโมง — แจ้งหน่วยบริการล่วงหน้าตามแผนสื่อสาร (ข้อ 7)

3รายการตรวจสอบ Cutover (Cutover Checklist)

3.1 ก่อน Cutover (Pre-Cutover)

  • ยืนยันสภาพแวดล้อม Production รพ.หลัก พร้อม (server, network, certificate, license)
  • ยืนยันแผน Rollback และทดสอบขั้นตอนถอยแล้ว (ข้อ 5)
  • แจ้งผู้ใช้/หน่วยบริการเรื่อง downtime และช่วง cutover แล้ว (ข้อ 7)
  • ทีม HosXP (ระบบเดิม) ยืนยัน standby ตลอดหน้าต่างเวลา + ช่องทางติดต่อ
  • ตรวจ Go/No-Go criteria ครบทุกข้อ (ข้อ 4)

3.2 ระหว่าง Cutover

  • Backup — Full backup ฐานข้อมูล + ไฟล์ HosXP สำเร็จและตรวจสอบ restore ได้
  • Freeze — ปิดรับ transaction ใหม่ใน HosXP + แจ้งหน่วยงานยุติการบันทึก
  • Migrate final — Migration delta รอบสุดท้าย HosXP→HA.OS เสร็จ ไม่มี error ค้าง
  • Reconciliation — เทียบยอดข้อมูลครบ 100% (จำนวนผู้ป่วย, IPD/เตียง, การเงิน, คลังยา, ผล LIS/PACS)
  • Smoke test — ทดสอบ flow หลักผ่าน (ลงทะเบียน, Admit, eMAR, จ่ายยา, ออกใบเสร็จ, เรียกฟิล์ม, ส่ง LIS)
  • Integration — เชื่อม สปสช. / PACS / LIS-RIS ทำงานถูกต้อง (ทดสอบ end-to-end)
  • เปิดให้ผู้ใช้ — ปลดล็อกระบบ HA.OS รพ.หลัก + ประกาศ Go-Live + แจ้งช่องทาง Hypercare

3.3 หลังเปิดใช้งานวันแรก

  • ตรวจ log/error วันแรก ไม่มี Critical/High ค้าง
  • ยืนยันธุรกรรมการเงิน/งานเรียกเก็บวันแรกตรงกับยอดจริง
  • สรุปประเด็นเข้า Hypercare (ข้อ 6) + ปิดประเด็นเร่งด่วน

5แผนถอยกลับ (Rollback Plan)

5.1 เงื่อนไขที่ต้องถอย (Rollback Trigger)

  • Reconciliation ไม่ครบ 100%/ข้อมูลคลาดเคลื่อนแก้ไม่ได้ในเวลา
  • พบ defect ระดับ Critical ที่กระทบบริการผู้ป่วย
  • Integration หลัก (เช่น สปสช. / การเงิน / PACS) ใช้งานไม่ได้

5.2 เวลาตัดสินใจถอย (Point of No Return) — ต้องตัดสินใจถอยภายใน __ ชั่วโมง หลัง migrate final และ ก่อน T-8 เปิดให้ผู้ใช้ · เวลาดำเนินการถอยให้เสร็จ (RTO) ภายใน __ ชั่วโมง

  • ยุติการใช้งาน HA.OS + แจ้งทุกหน่วยงาน
  • กู้คืน HosXP (ระบบเดิม) จาก backup (T-3) และตรวจ restore ครบ
  • ปลด Freeze HosXP + เปิดให้บันทึกต่อ
  • บันทึกสาเหตุ + วางแผน cutover รอบใหม่
