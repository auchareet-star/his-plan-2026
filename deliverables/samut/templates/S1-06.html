1วัตถุประสงค์และขอบเขตการทดสอบ

1.1 วัตถุประสงค์

  • ยืนยันว่าระบบ HIS เมืองสมุทร ทำงานได้ตรงตามข้อกำหนด (1,358 ข้อกำหนด ใน 36 โมดูล) และเอกสารออกแบบในงวด 2
  • ตรวจสอบการเชื่อมต่อ (Integration) ภายใน Smart IPD (Doctor / Nurse / Pharmacist Module, eMAR, ER, วิสัญญี, เวชศาสตร์ฟื้นฟู) และระบบที่เกี่ยวข้องทำงานถูกต้องครบถ้วน
  • ยืนยันความถูกต้องของการโยกย้ายข้อมูลจากระบบเดิม HosXP → HA.OS (reconciliation 100%)
  • ลดความเสี่ยงของข้อบกพร่อง (Defect) ระดับ Critical/High ก่อนขึ้นใช้งานจริง (Go-live รพ.หลัก งวด 2)

1.2 ขอบเขต (In-scope)

  • โมดูล HIS ตามสัญญาแนบท้าย 1 ครบ 36 โมดูล (อาทิ คลังยาและวัสดุ 101, Doctor Module 91, Nurse Module 84, งานเรียกเก็บ 81, Queue & Patient Self Visit 74)
  • Smart IPD: Doctor/Nurse/Pharmacist Module, eMAR, Master Data Smart IPD, ER, ห้องตรวจสุขภาพ, เวชศาสตร์ฟื้นฟู, วิสัญญีพยาบาล
  • การทดสอบการโยกย้ายข้อมูล (Migration Test) จาก HosXP (ระบบเดิม) → HA.OS รวม Dress Rehearsal 1–2 รอบ
  • จุดเชื่อมต่อทุก Interface และการใช้งานครบ 9 จุดบริการ (ปากน้ำ 3 จุด + ปู่เจ้าฯ 6 จุด)

1.3 นอกขอบเขต (Out-of-scope)

  • ระบุงานที่ไม่รวมในแผนทดสอบ เช่น โครงสร้างเครือข่าย/ฮาร์ดแวร์ของโรงพยาบาล, บริการ AI/Speech-to-Text (คิดค่าใช้จ่ายแยก) ฯลฯ

2ระดับการทดสอบและเกณฑ์เริ่ม/จบ (Test Levels & Entry/Exit Criteria)

ระดับขอบเขต/ผู้รับผิดชอบเกณฑ์เริ่ม (Entry)เกณฑ์จบ (Exit)
Unit Testฟังก์ชันย่อย / Dev (ซี/มิน/โอปอล์)โค้ดผ่าน build, มี test caseผ่าน ≥ 80% coverage, ไม่มี bug บล็อก
SIT
(System Integration)		การเชื่อมต่อระหว่างโมดูล / Tester Lead + DevUnit ผ่าน, ติดตั้งบน Test env, Interface พร้อมทุก Interface ส่ง-รับข้อมูลถูกต้อง, ไม่มี Critical/High ค้าง
Migration Testโยกย้าย HosXP → HA.OS / Senior Imp เตทีม (HosXP) นำMapping ครบ, มี Dress Rehearsal env พร้อมReconciliation 100%, จำนวน/ยอดข้อมูลตรงกัน
UAT
(User Acceptance)		ผู้ใช้จริงตามกระบวนงาน / Key User + TesterSIT ผ่าน, มี UAT script, ข้อมูลทดสอบพร้อมผู้ใช้ลงนามรับรอง, ไม่มี Critical/High ค้าง
E2E
(End-to-End)		กระบวนงานเต็มเส้น (ผู้ป่วยเข้า→ตรวจ→IPD→จ่ายยา→เบิกจ่าย) / Testerโมดูลที่เกี่ยวข้องผ่าน SITทุก business flow หลักผ่านครบ
Performanceโหลด/ความเร็ว/ความทนทาน / SA + Testerระบบเสถียรบน env ใกล้เคียง Productionผ่านเกณฑ์ตอบสนอง ≤ 3 วิ ที่ __ users พร้อมกัน
Securityช่องโหว่/สิทธิ์เข้าถึง/PDPA / SA + Testerระบบติดตั้งครบ, มี checklist OWASPไม่พบช่องโหว่ระดับ High ขึ้นไป

SIT จะทดสอบกรณีแพทย์สั่งยาใน Doctor Module แล้วส่งต่อไป Pharmacist Module และบันทึกการให้ยา (eMAR) ใน Nurse Module ให้แสดงผลถูกต้องครบเส้นทาง

3นิยามความรุนแรงของข้อบกพร่อง (Defect Severity) และ SLA การแก้ไข

ระดับนิยามSLA แก้ไข (เริ่ม/แก้เสร็จ)ผลต่อ Go-Live
Criticalระบบหยุดทำงาน/ข้อมูลเสียหาย/กระทบความปลอดภัยผู้ป่วย — ไม่มีทางเลี่ยงเริ่มภายใน 2 ชม. / workaround ภายใน 8 ชม.บล็อก ห้ามขึ้นระบบ
Highฟังก์ชันหลักใช้งานไม่ได้ มีทางเลี่ยงยากภายใน 1 วันทำการบล็อก
Mediumฟังก์ชันรองผิดพลาด มีทางเลี่ยงได้ภายใน 3 วันทำการไม่บล็อก (มีแผนแก้)
Lowข้อบกพร่องเล็กน้อย/ความสวยงาม/ข้อความภายใน งวดถัดไปไม่บล็อก
⚠️ เกณฑ์ Go-Live: ต้องไม่มี Defect ระดับ Critical และ High ค้างอยู่ จึงจะผ่าน Exit ของ UAT และอนุมัติขึ้นระบบจริง · SLA Critical อ้างอิงบันทึกเพิ่มเติม 26 มิ.ย. 2569 (เริ่มแก้ภายใน 2 ชม. + workaround ภายใน 8 ชม.)

4สภาพแวดล้อมและข้อมูลทดสอบ (Test Environment & Data)

4.1 สภาพแวดล้อม

สภาพแวดล้อมวัตถุประสงค์หมายเหตุ/สเปก
DEVพัฒนา + Unit Test________
SIT / TESTทดสอบเชื่อมต่อ + E2Eแยกจาก Production, มี mock ระบบที่เกี่ยวข้อง
Migration / Dress Rehearsalซ้อมโยกย้าย HosXP → HA.OSสำเนาข้อมูลจริง HosXP (ปกปิด) สำหรับซ้อม 1–2 รอบ
UAT / Stagingผู้ใช้ทดสอบ + Performanceสเปกใกล้เคียง Production

4.2 ข้อมูลทดสอบ (Test Data)

  • ใช้ข้อมูลจำลอง (Synthetic) สำหรับเคสทั่วไป และข้อมูลจริงที่ปกปิด (Masked/De-identified) สำหรับ Migration ตาม PDPA
  • เตรียมชุดข้อมูลผู้ป่วยตัวอย่างครอบคลุมเคส OPD/IPD/ER (admit, สั่งยา, ให้ยา eMAR, ผ่าตัด/วิสัญญี, เวชศาสตร์ฟื้นฟู, เรียกเก็บ)
  • ผู้รับผิดชอบเตรียมข้อมูล: Senior Imp เตทีม (HosXP) + ทีม Dev

เตรียมผู้ป่วย IPD จำลอง 1 ราย ที่มี admit + แพทย์สั่งยา + เภสัชจ่ายยา + พยาบาลบันทึก eMAR + เรียกเก็บ เพื่อทดสอบ flow ครบทั้งระบบ
