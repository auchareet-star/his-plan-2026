1กลยุทธ์การพัฒนา (Development Strategy)

  • พัฒนาแบบ รายโมดูลเป็นเฟส (Phased / Iterative) ส่งมอบทีละชุดเพื่อให้ทดสอบและตรวจรับได้เร็ว
  • ยึดหลัก Config ก่อน Custom ก่อน Dev ใหม่ — ใช้ความสามารถมาตรฐานของ HA.OS ก่อน ปรับแต่ง (customize) เท่าที่จำเป็น เขียนใหม่เฉพาะส่วน Gap (ต่อยอดจาก Fit/Gap)
  • จัดลำดับตาม dependency และความสำคัญต่อ Go-live (โมดูลแกน HIS + Smart IPD ก่อน)
  • ใช้ระบบควบคุมเวอร์ชันและ CI/CD เพื่อ deploy ขึ้น Test/Staging อัตโนมัติ
หลักการจัดประเภทงาน: CONFIG ตั้งค่าได้ทำก่อน, CUSTOM ปรับแต่งบนฐาน HA.OS, DEV ใหม่ ที่ซับซ้อนหรือพึ่งระบบฐานจาก DTMU Phase 2 วางไว้ให้สอดคล้องกับช่วงที่ระบบฐานพร้อม

4นิยามความเสร็จสมบูรณ์ (Definition of Done)

  • พัฒนา/Config ครบตามความต้องการ (SRS) ที่ sign-off แล้ว
  • ผ่าน Unit Test / Code Review
  • ผ่าน Integration Test กับโมดูล/ระบบที่เกี่ยวข้อง
  • ไม่มี defect ระดับ Critical/High คงค้าง
  • มีเอกสารประกอบ (คู่มือ/หมายเหตุการตั้งค่า) ครบ
  • Deploy ขึ้น Staging และพร้อมให้ผู้ใช้ทดสอบ UAT
  • เจ้าของงานยืนยันยอมรับฟังก์ชัน (Acceptance)

5การเชื่อมต่อระบบ (Integration) และโมดูลที่รอ DTMU Phase 2

การเชื่อม สปสช./HL7 และบางโมดูลพึ่งระบบฐานจาก DTMU Phase 2 (ระบบฐาน) เป็นช่วง ๆ จึงต้องลดการพึ่งพาแบบ blocking ด้วย mock/stub

  • ออกแบบ Interface ตามสัญญา/สเปก (สปสช./HL7) ก่อนระบบฐาน DTMU Phase 2 พร้อม
  • สร้าง Mock/Stub ของ Interface และระบบฐาน เพื่อให้ Dev พัฒนา/ทดสอบต่อได้ระหว่างรอ DTMU Phase 2
  • เมื่อระบบฐาน DTMU Phase 2 พร้อม (Consult Sprint) จึงทำ Integration Test กับระบบจริงและปรับแก้
  • บันทึกผลทดสอบ interface และ sign-off ร่วมกับ Product Specialist DTMU
⚠️ จุดวิกฤต: งาน Integration ที่พึ่งระบบฐาน DTMU Phase 2 ต้องจองคิวล่วงหน้าและทำเป็น batch — ใช้ mock/stub ครอบไว้เพื่อไม่ให้ Dev หยุดรอ
Dev ──▶ HA.OS ──▶ [ Mock/Stub Interface + ระบบฐาน ] (ระหว่างรอ DTMU Phase 2) │ Consult Sprint ───┴──▶ [ DTMU Phase 2 จริง + สปสช./HL7 ] (Integration Test + sign-off)
