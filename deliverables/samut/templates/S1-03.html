3วิธีการโอนย้าย (ETL: Extract / Transform / Load)

3.1 Extract

ดึงข้อมูลจาก HosXP เข้า Staging โดยไม่กระทบระบบ Production (read-only / นอกเวลาทำการ) — Senior Imp เตทีม (HosXP) เป็นผู้นำการ Extract เพราะคุ้นโครงสร้างฐานข้อมูล HosXP

3.2 Transform

  • Mapping ฟิลด์ต้นทาง HosXP → ปลายทาง HA.OS (ตาราง mapping แยกไฟล์)
  • แปลงรูปแบบ: ปี พ.ศ. → ค.ศ., รหัสยา/วินิจฉัย (ICD/TMT), encoding ภาษาไทย
  • ทำความสะอาดข้อมูล (Data Cleansing) — ค่าซ้ำ/ว่าง/ไม่ valid, HN ซ้ำข้ามจุดบริการ
  • ตรวจ Referential Integrity ก่อน Load

3.3 Load

นำเข้า HA.OS ตามลำดับ master → transaction พร้อม log การนำเข้าและ error report

HosXP DB ──(Extract)──▶ Staging ──(Transform/Cleanse)──▶ Validated ──(Load)──▶ HA.OS DB │ │ └──────────────────────── Reconciliation (ข้อ 5) ◀────────────────────────┘

Dress Rehearsal #1__/__/2026ทดสอบ pipeline + จับ error mapping เบื้องต้น
Dress Rehearsal #2__/__/2027ทดสอบเต็มชุด + วัดเวลาประมวลผล + reconciliation 100%
Dress Rehearsal #3 (สำรอง)__/__/2027ยืนยันแก้ไขข้อบกพร่องครบ (ถ้าจำเป็น)
Cut-over (รอบจริง)__/__/2027โอนจริงก่อน Go-live รพ.หลัก, reconciliation 100%
เกณฑ์: ต้องทำ Dress Rehearsal อย่างน้อย 1–2 รอบ และผ่าน reconciliation 100% ก่อนจึงจะอนุมัติ Cut-over รอบจริง

5เกณฑ์การตรวจสอบความถูกต้อง (Reconciliation 100%)

  • นับจำนวน record ต้นทาง = ปลายทาง ทุก dataset (Record Count = 100%)
  • ตรวจยอดรวมเชิงตัวเลข (Control Total) เช่น ยอดลูกหนี้/เคลม สปสช. รวม
  • สุ่มตรวจรายตัว (Sampling) อย่างน้อย __% หรือ __ records/dataset
  • ตรวจ Referential Integrity (ไม่มี orphan record)
  • ตรวจฟิลด์สำคัญ: HN, ชื่อ-สกุล, วันเกิด (พ.ศ.→ค.ศ.), สิทธิการรักษา (สปสช.)
  • เจ้าของงานรายโมดูลยืนยัน (sign-off) ผลการ reconcile

8ความเสี่ยงและแผนถอยกลับ (Rollback)

ความเสี่ยงโอกาสผลกระทบการรับมือ
