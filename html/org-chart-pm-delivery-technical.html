Project Manager (Delivery) · (Technical) · Project Coordinator

ครอบคลุมทั้ง 2 โครงการ: DTWU/วลัยลักษณ์ และ เมืองสมุทร (สมุทรเวชกิจ + ปู่เจ้าฯ) — มีน ดูแลฝั่ง Delivery ทั้ง 2 ไซต์, อาร์ต ดูแลฝั่ง Technical ทั้ง 2 ไซต์, โม ดูแลฝั่งประสานงาน/ติดตามงาน (PC) ทั้ง 2 ไซต์

Delivery PM — มีน Technical PM — อาร์ต PC — โม

โครงสร้างองค์กร

Delivery PM

มีน
  • Project Plan / Org Chart / RACI
  • Timeline & Sprint ทั้ง 2 ไซต์ (PC ดูแล Tracker จริง)
  • Kick-off & Weekly Status กับลูกค้า
  • งวดส่งมอบ (Milestone 1–4) + ขอ Sign-off
  • Risk & Issue เชิงสัญญา/ธุรกิจ
  • Change Request
  • UAT Planning + Key User Coordination
  • Training Coordination
  • Go-live Coordination + Cut-over/Failover (ฝั่งประสาน)
  • Hypercare / War Room (PC ตั้ง War Room จริง)
  • Project Closure

Technical PM

อาร์ต
  • Fit Gap Analysis เชิงเทคนิค (Fit/Config/Partial/Gap)
  • SRD / Technical Design Review
  • API / Interface (SAP, WU-HRMS)
  • Data Migration (HosXP → ระบบใหม่)
  • Dev Scope / Config โมดูลใหม่
  • Infrastructure & Deployment Planning
  • Security Review
  • Technical Risk & Root Cause Analysis
  • Production Support / Monitoring หลัง Go-live
  • Stabilization / ติดตาม KPI ระบบ

Project Coordinator (PC)

โม
  • Master Sprint Tracker ทั้ง 2 ไซต์ — ติดตามสถานะทุก Epic
  • จัดตารางนัด Workshop Fit Gap ร่วม Process Owner/DTMU
  • ประสานห้องประชุม/นัดหมายทุกฝ่าย
  • รวบรวม Go-live Readiness Checklist จากทุกทีม (Dev/Imp/Training/Infra)
  • ตั้ง War Room วัน Go-live + ติดตาม Incident แบบเรียลไทม์
  • ติดตาม Stabilization / Issue Tracker หลัง Go-live
  • สรุปรายงานสถานะประจำสัปดาห์ให้ Delivery PM + Technical PM
  • Consult งานวิเคราะห์/Fit Gap เพิ่มเติม (พื้นฐาน HIS ตรง + เคยทำงาน รพ.)

ทีมวิเคราะห์ / Get Requirement (ทีมเดียว ใช้ร่วมกันทั้ง DTWU และเมืองสมุทร)

ทำไมต้องเป็นทีมเดียว: ทั้ง DTWU/วลัยลักษณ์ และเมืองสมุทร/ปู่เจ้า เอาระบบ DTMU มาพัฒนาต่อยอด เป็นฐานเดียวกัน — ต้องรวบรวมความต้องการจาก 2 ไซต์มา Fit Gap ร่วมกับทีม DTMU (ผู้เชี่ยวชาญระบบเดิม) เพื่อให้ผลตัดสินใจ Fit/Config/Partial/Gap ใช้ได้กับทุกไซต์โดยไม่ชนกันเอง — ถ้าแยกทีมวิเคราะห์ตามไซต์ เสี่ยงตัดสินใจขัดแย้งกันบนระบบฐานเดียวกัน

R — ลงมือทำหลัก: PO จูน + BA ชมพู่ ทำ Fit Gap และเก็บ Requirement ร่วมกันทั้ง 2 ไซต์ ไม่ได้แยกตามไซต์ของตัวเอง

C — ที่ปรึกษาเสริม: เตทีม (ความรู้ HosXP/Data Migration เฉพาะทาง) และ โม (แม้ตำแหน่งหลักคือ PC แต่มีพื้นฐาน HIS ตรงและเคยทำงาน รพ. มาก่อน จึงช่วยวิเคราะห์และร่วมตัดสินใจบางเรื่องได้ ไม่ใช่แค่ Support ธุรการ)

C — ที่ปรึกษาระดับผลิตภัณฑ์: มีน (PO) ให้ความเห็นจากฐาน DTMU/HA.OS แยกบทบาทจากทีมวิเคราะห์หน้างาน
