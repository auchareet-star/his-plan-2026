1ภาพรวมและความสำคัญ

การโอนย้ายข้อมูลจาก HosXP (ระบบเดิม) มายัง HA.OS (ระบบใหม่) เป็นความเสี่ยงสูงสุดของโครงการ เพราะข้อมูลกระจายอยู่ 3 แหล่ง/3 เวอร์ชัน ที่ schema ต่างกัน มีผู้ป่วยซ้ำข้ามแหล่ง และมีข้อมูลที่กระทบการรักษาโดยตรง (ประวัติแพ้ยา, ยาที่ใช้ต่อเนื่อง) หากผิดพลาดอาจเป็นอันตรายต่อผู้ป่วย

🔴 หลักการสำคัญ: "ถูกต้อง 100% สำหรับข้อมูลวิกฤต" มาก่อน "ครบทุกข้อมูล" · ทำ Data Profiling → Cleansing → Dedup ก่อนซ้อม · ซ้อม ≥ 3 รอบ (แยกทีละแหล่งก่อนรวม) · มี Cutover Runbook + เกณฑ์ Go/No-Go + จุด Rollback ก่อนรอบจริง

ระบบปลายทางHA.OS (ตาราง TB_* ธุรกรรม / TBM_* master)

2แหล่งข้อมูลต้นทาง — 3 แหล่ง SOURCE

ต้อง dedup ข้าม 3 แหล่ง เพราะผู้ป่วยรายเดียวอาจมีในหลายเวอร์ชัน (โดยเฉพาะ v4 ↔ v3)

แหล่งครอบคลุมลักษณะ / ความท้าทาย
v4 (หลัก)IPD + OPD รพ.หลักฐานหลัก · schema เฉพาะ IPD ต้อง map แยก · ข้อมูลปริมาณมากสุด
v3 (OPD)ผู้ป่วยนอก (เวอร์ชันเก่า)schema ต่างจาก v4 · คนไข้อาจซ้ำกับ v4 → ต้อง match/merge
vPCUคลินิกย่อย / PCUฐานที่ 3 · ต้อง dedup ข้าม 3 แหล่ง · รหัส/มาตรฐานอาจต่าง
💡 แตกงานเป็น 3 workstream แยก (v4 / v3 / vPCU) เพื่อ Extract + Profiling + Cleansing ขนานกัน แล้วจึงรวม (merge) ในขั้น Dedup

3ระบบปลายทาง HA.OS TARGET

โครงสร้าง HA.OS แบ่งเป็นตาราง master (TBM_* เช่น ICD10, กลุ่มแพ้ยา, รายการค่าใช้จ่าย) และตาราง ธุรกรรม (TB_* เช่น ผู้ป่วย, การรับบริการ, คำสั่ง, ใบเสร็จ)

⚠️ ข้อควรระวัง: ต้อง ตั้งค่า Master Data / Lookup ปลายทางให้ครบก่อน (สิทธิการรักษา, รหัสยา, ICD10, รายการค่าบริการ) ไม่งั้น Load ธุรกรรมจะ orphan / อ้างอิงไม่เจอ

5วิธีการโอนย้าย (ETL)

Extract→ Data Profiling→ Cleansing→ Dedup / Merge→ Transform + Map→ Load (HA.OS)→ Verify / Reconcile
ขั้นสาระสำคัญ
Extractดึงข้อมูลจาก v4 / v3 / vPCU แยกกัน · snapshot วันที่ตัดยอดชัดเจน · เก็บใน staging area
Data Profilingวิเคราะห์คุณภาพข้อมูลต้นทาง (null, รูปแบบผิด, ค่าซ้ำ, รหัสไม่อยู่ใน master) → สรุป Data Quality Issues
Cleansingแก้/เติมข้อมูลตามกฎ (format บัตรปชช./ชื่อ, รหัสสิทธิ, หน่วยยา) · บันทึกกฎการแปลงทุกข้อ
Dedup / Mergeจับคู่ผู้ป่วยข้าม 3 แหล่ง → สร้าง Golden Record (ดูข้อ 6)
Transform + Mapแปลง schema HosXP → HA.OS ตาม Data Mapping (ข้อ 4) · map lookup/รหัสมาตรฐาน
LoadLoad ตามลำดับ: Master/Lookup → ผู้ป่วย → visit → orders → การเงิน (ตาม dependency)
VerifyReconciliation (ข้อ 8) · Key User sampling · เทียบยอด/จำนวน

6Data Cleansing & Deduplication (หัวใจของงาน)

กลยุทธ์จับคู่ผู้ป่วย (Patient Matching)

  • Exact match: เลขบัตรประชาชน (ตัวหลัก) · HN เดิม
  • Fuzzy match: ชื่อ-สกุล + วันเกิด + เพศ (กรณีบัตรปชช.ว่าง/ผิด) → ตั้ง threshold + คนตรวจยืนยัน
  • Golden Record: เลือกข้อมูลล่าสุด/สมบูรณ์สุดเป็นหลัก · รวมประวัติจากทุกแหล่ง · ประวัติแพ้ยาให้รวมทุกแหล่ง (union) ห้ามตกหล่น
⚠️ กรณีกำกวม (match ไม่ชัด) ให้เข้าคิว manual review โดย Key User เวชระเบียน ห้าม auto-merge ข้อมูลการแพทย์ที่เสี่ยง

ตัวอย่างกฎ Cleansing

  • รหัสสิทธิการรักษา → map เป็นรหัสมาตรฐาน สปสช. ปลายทาง
  • รหัสยา/หน่วย → map เข้า drug master HA.OS (ATC/รหัสโรงพยาบาล)
  • ICD10 → ตรวจกับ master ICD10 (มีไฟล์เทียบ icd_compare) · รหัสไม่ผ่านให้ flag

  • ตรวจ mapping CSOP/SSOP (มีไฟล์ csop_ssop_mapping.xlsx) และการส่งออก 42/43 แฟ้ม
  • เทียบ ICD10 ต้นทางกับ master (ไฟล์ icd_compare_pdf_vs_tbm_icds) · รหัสไม่ผ่าน flag แก้ก่อน Go-live
  • ทดสอบ ส่งเบิก e-Claim จริง (CSOPBIL) จากข้อมูลที่ย้ายแล้ว → ยืนยันไฟล์ผ่านการตรวจของ สปสช. (ดูตัวอย่างใน check-claim/)
  • เกณฑ์: การเบิกจ่าย/รายได้หลัง Go-live ต้องไม่สะดุด

10Cut-over (รอบจริง) & Rollback

Cutover Runbook + Go/No-Go

  1. Freeze / ตัดยอด HosXP เดิม (แจ้งล่วงหน้า · กำหนดหน้าต่าง downtime นอกเวลาให้บริการหลัก)
  2. Extract snapshot สุดท้าย → รัน pipeline ที่ผ่านการซ้อมรอบ 3
  3. Load + Reconciliation ข้อมูลวิกฤต 100%
  4. ประชุม Go/No-Go ตามเกณฑ์ที่ตั้งไว้ล่วงหน้า (ไม่มี issue Critical/High ค้าง)
  5. เปิดใช้ HA.OS · เฝ้าระวังใกล้ชิด (Hypercare)
🔴 Rollback plan: กำหนดจุดถอยกลับและเกณฑ์ตัดสินใจ Rollback ล่วงหน้า · หากพบปัญหาวิกฤตในหน้าต่าง cut-over → กลับไปใช้ HosXP เดิม (ยัง read/write ได้) โดยไม่สูญข้อมูล · ทดสอบ Rollback ในซ้อมรอบ 3
⚠️ หน้าต่าง Downtime: เลือกช่วงผู้ป่วยน้อย · เตรียมแบบฟอร์มสำรอง (manual) ระหว่าง downtime · สื่อสารทุกจุดบริการล่วงหน้า
